<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DB write path → SQLite | openrouter_provider_preference value flows from application layer to SQLite via parameterized query |
| DB read path → Account type | Column value flows from SQLite row into AccountRow → Account → AccountResponse via toAccount()/toAccountResponse() |
| JSON parse in toAccountResponse() | Stored JSON string is parsed in the application layer — malformed JSON is caught and returns null |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02-01 | Tampering | setOpenrouterProviderPreference() SQL write | mitigate | Parameterized query `UPDATE accounts SET openrouter_provider_preference = ? WHERE id = ?` with `[preference, accountId]` array — prevents SQL injection (ASVS V5.3.4). No string interpolation. |
| T-03-02-02 | Tampering | JSON value stored opaque — no server-side execution | accept | The string is stored as TEXT and later parsed to string[] only in toAccountResponse(). No eval(), no exec. Input validation (valid JSON, valid provider names, length limits) is deferred to Phase 5 (PATCH endpoint) per RESEARCH.md security domain analysis. |
| T-03-02-03 | Information Disclosure | openrouterProviderPreference in AccountResponse | accept | Field contains provider routing preferences (e.g., ["openai", "anthropic"]) — no PII, no secrets. Same exposure level as existing modelMappings field. |
| T-03-02-04 | Denial of Service | Malformed JSON in openrouter_provider_preference crashes toAccountResponse() | mitigate | try/catch guard around JSON.parse in toAccountResponse() — malformed JSON returns null (same pattern as modelMappings, line 335-354 of account.ts). |
| T-03-02-05 | Tampering | Migration alters existing accounts table | mitigate | idempotency guard `if (!initialAccountsColumnNames.includes("openrouter_provider_preference"))` prevents double-apply. `TEXT DEFAULT NULL` ensures existing rows receive NULL (no data loss). Migration runs inside `migrationTx` transaction — failure rolls back. |
</threat_model>
